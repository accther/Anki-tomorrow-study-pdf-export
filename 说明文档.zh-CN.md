# Anki 明日复习 PDF 插件说明

这是一份面向“会用 Anki，但不一定懂代码”的说明文档。

它回答 4 个问题：

1. 这个插件用了什么技术栈？
2. 它是怎么工作的？
3. 每个文件分别负责什么？
4. 文档里出现的专业术语，普通人该怎么理解？


## 1. 这个插件是做什么的？

这个插件的目标很直接：

- 从 Anki 里整理出“明天要学的卡片”
- 按 Anki 实际学习顺序导出成 PDF
- 方便打印出来，在纸上做题、标记，再回到 Anki 同步

这个插件不是简单地“把某个牌组全部打印出来”。
它真正做的是：

- 先判断明天哪些卡片会出现
- 再尽量按照 Anki 调度器的实际出卡顺序取卡
- 最后把这些卡片排版成适合打印的 PDF


## 2. 技术栈是什么？

### 2.1 Python

专业名词：`Python`

通俗解释：
这是插件的主要编程语言。Anki 的插件系统本身就是基于 Python，所以插件逻辑基本都写在 Python 里。

本插件里，Python 负责：

- 打开导出窗口
- 读取牌组范围
- 复制临时数据库快照
- 调用 Anki 的调度器
- 组织卡片数据
- 触发 PDF 渲染


### 2.2 Anki Add-on API

专业名词：`Anki Add-on API`

通俗解释：
可以理解为“Anki 官方留给插件调用的一组内部接口”。
插件不是自己重新做一个背单词软件，而是借用 Anki 已有的能力。

本插件主要调用了这些能力：

- `mw.form.menuTools.addAction(...)`
  - 往 Anki 的 `Tools` 菜单里加一个按钮
- `QueryOp(...).run_in_background()`
  - 把耗时任务放到后台执行，避免界面卡住
- `Collection(...)`
  - 打开一个 Anki 数据库
- `col.sched`
  - 使用 Anki 调度器获取卡片队列
- `mw.prepare_card_text_for_display(...)`
  - 用 Anki 自己的方式预处理卡片内容


### 2.3 SQLite

专业名词：`SQLite`

通俗解释：
Anki 的卡片、笔记、牌组，本质上都存放在一个本地数据库文件里。
这个数据库格式叫 SQLite。

本插件里，SQLite 的作用不是“长期存数据”，而是：

- 对当前 Anki 数据库做一个临时快照
- 在快照上模拟“明天”
- 保证正式数据库不被改动

这点非常重要。
插件不是直接在你正在用的正式库上乱改，而是先复制一份临时副本，再在副本上计算。


### 2.4 PyQt / Qt

专业名词：`PyQt6`、`Qt`

通俗解释：
这是 Anki 桌面界面的底层 GUI 框架。
按钮、弹窗、树形列表、文件保存窗口，本质上都来自 Qt。

本插件用 Qt 来做：

- 牌组选择窗口
- 勾选框
- PDF 文件保存窗口
- 隐藏的网页渲染视图


### 2.5 WebView + HTML + CSS + JavaScript

专业名词：`WebView`、`HTML`、`CSS`、`JavaScript`

通俗解释：
虽然最终导出的是 PDF，但中间其实先把卡片排成一个网页，然后再把这个网页“打印”为 PDF。

可以把它理解成：

1. 先做一个临时网页
2. 网页里放入卡片内容
3. 用网页排版能力控制字体、空白、分页
4. 最后导出成 PDF

为什么这么做？
因为 Anki 的卡片本身就带有：

- HTML 结构
- CSS 样式
- JavaScript 脚本
- MathJax / LaTeX 公式渲染

如果直接用纯文本方式生成 PDF，很多复杂卡片会失真。
而网页渲染方式更接近 Anki 预览和复习界面的真实显示效果。


### 2.6 MathJax / LaTeX

专业名词：`LaTeX`、`MathJax`

通俗解释：
这是数学公式显示相关的技术。

- `LaTeX` 像是一种“写公式的语言”
- `MathJax` 像是一个“把公式语言翻译成可显示排版”的前端引擎

比如你写：

```text
\sin \alpha
```

最终会显示成数学公式，而不是原始代码。

本插件为了尽量接近 Anki 预览，专门保留了公式渲染链路。


## 3. 这个插件的整体实现原理

可以把整个流程理解成 5 步。


### 第一步：用户选择范围

对应文件：
[`__init__.py`](./__init__.py)

插件先弹出一个窗口，让你选择：

- 哪些牌组要导出
- 是否包含子牌组
- 是否包含复习卡 / 学习卡
- 是否包含明天会出现的新卡

这里看到的牌组树，就是从 Anki 当前牌组结构里读出来的。


### 第二步：创建临时数据库快照

对应文件：
[`scheduler_snapshot.py`](./scheduler_snapshot.py)

这是最核心的一步。

插件不会直接操作你正在使用的正式数据库，而是：

1. 找到当前 `collection.anki2`
2. 用 SQLite 的 `backup()` 机制复制出一个临时数据库
3. 后续所有“明天模拟”都在这份临时副本上完成

专业名词：`数据库快照（snapshot）`

通俗解释：
就像给当前书桌拍了一张照片，后面在照片上做标记，不会把真实书桌弄乱。


### 第三步：把临时库“推进到明天”

对应文件：
[`scheduler_snapshot.py`](./scheduler_snapshot.py)

插件不会真的等到明天，而是通过修改临时数据库中的时间相关字段，模拟“明天已经到了”。

核心思路是：

- 把 collection 的基准日期往前挪一天
- 把部分学习卡的绝对时间戳也同步调整

这样做的结果是：
那些“明天才到期”的卡片，在临时库里会变成“现在就该出现”。

专业名词：`时间偏移（time shift）`

通俗解释：
不是修改现实时间，而是把“考试日历”在临时副本里往前拨一天。


### 第四步：按 Anki 调度器顺序取卡

对应文件：
[`scheduler_snapshot.py`](./scheduler_snapshot.py)

这是插件最重要的设计目标之一：
尽量和 Anki 实际学习顺序一致。

插件优先尝试：

- 直接调用 Anki 调度器的队列接口 `get_queued_cards()`

如果这个接口在某些版本不可用，再退回到：

- SQL 兜底排序

为什么优先走调度器？
因为真正决定出卡顺序的，不只是“到期时间”，还可能包括：

- 新卡顺序
- 复习卡顺序
- bury 规则
- 牌组选项
- 学习队列规则

这些复杂规则，Anki 调度器本身最清楚。

专业名词：`Scheduler`

通俗解释：
它就像 Anki 的“出题总控台”，决定下一张该给你看哪张卡。


### 第五步：把卡片渲染成网页，再打印成 PDF

对应文件：
[`pdf_renderer.py`](./pdf_renderer.py)

导出 PDF 时，并不是直接把字符串塞进 PDF。
真实流程是：

1. 先把所有卡片拼成一份 HTML 页面
2. 给这份页面套上 CSS 样式
3. 让 JavaScript、MathJax、模板脚本执行
4. 等公式、加密卡片、样式渲染完成
5. 最后调用 `printToPdf()`

这样做的优势是：

- 更接近 Anki 预览的显示效果
- 更容易正确显示公式
- 更容易兼容图片、模板样式、脚本型卡片


## 4. 三个核心文件分别做什么？

### 4.1 [`__init__.py`](./__init__.py)

这是插件的“入口文件”。

你可以把它理解成“前台总控”。

它主要负责：

- 注册菜单项 `Export Tomorrow to PDF`
- 弹出牌组选择窗口
- 收集用户勾选的参数
- 让导出任务在后台执行
- 在任务结束后弹出成功或失败提示

如果把整个插件比作一家店：

- `__init__.py` 就像前台和总接待
- 它不负责真正做菜，但负责接单、分发任务、回传结果


### 4.2 [`scheduler_snapshot.py`](./scheduler_snapshot.py)

这是插件的“调度与数据核心”。

它主要负责：

- 创建 SQLite 临时快照
- 在快照里模拟明天
- 设置当前活动牌组范围
- 尝试从 Anki 调度器读取队列
- 把卡片整理成导出所需的数据结构

这是最接近“业务核心逻辑”的文件。

如果把插件比作一家店：

- 它像后厨
- 真正决定“今天要处理哪些食材、按什么顺序上菜”


### 4.3 [`pdf_renderer.py`](./pdf_renderer.py)

这是插件的“排版与输出核心”。

它主要负责：

- 生成导出页面的 HTML
- 写页面样式 CSS
- 启动 WebView 渲染
- 等待 MathJax 和脚本执行完成
- 导出为 PDF

它不负责判断“哪些卡应该导出”，只负责“这些卡如何被印出来”。

如果把插件比作一家店：

- 它像印刷车间
- 负责把已经准备好的内容印刷成最终成品


## 5. 关键数据结构是什么意思？

这些是代码里最重要的几个“数据盒子”。


### 5.1 `ExportRequest`

所在文件：
[`scheduler_snapshot.py`](./scheduler_snapshot.py)

专业解释：
一个不可变的数据对象，用来描述本次导出的输入条件。

通俗解释：
就是“这次导出订单”。

里面记录了：

- 选中了哪些牌组
- 是否包含子牌组
- 是否包含新卡
- 是否包含复习卡
- 目标学习日是哪一天


### 5.2 `QueuedCardForPdf`

所在文件：
[`scheduler_snapshot.py`](./scheduler_snapshot.py)

专业解释：
一个导出中间模型，表示“已经排好顺序、可以进入 PDF 的单张卡片”。

通俗解释：
就是“已经加工完成、准备排版的一张题”。

里面会保存：

- 这是第几张
- 卡片 ID
- 笔记 ID
- 牌组 ID
- 卡片类型
- 正面 HTML
- 背面 HTML


### 5.3 `TomorrowExportResult`

所在文件：
[`scheduler_snapshot.py`](./scheduler_snapshot.py)

专业解释：
整个导出任务的最终结果对象。

通俗解释：
可以理解为“打包好的整批导出内容”。

它包含：

- 所有待导出的卡片
- 导出时间
- 目标学习日
- 所选牌组名称
- 页面尺寸


## 6. 这个插件如何保证“尽量不伤库”？

这个问题很关键。

插件为了减少对正式库的影响，做了这些设计：

- 不在正式数据库上模拟明天
- 先创建 SQLite 临时快照
- 在临时快照上做时间偏移和调度读取
- 导出完成后关闭并删除临时目录

专业名词：`隔离式计算`

通俗解释：
就像先复印一份试卷，在复印件上写答案，不在原卷上乱写。

不过要实话实说：
只要插件涉及数据库、调度器、网页渲染，就属于“中高复杂度插件”。
它不是一个纯 UI 小插件。

所以工程上要遵守一个原则：

- 以后如果继续改功能，优先小步修改、小步测试
- 尤其不要轻易改数据库复制逻辑


## 7. 为什么这个插件实现起来比看上去复杂？

因为你的需求表面上像“打印卡片”，其实背后有 3 个难点。


### 难点一：不是打印全部，而是打印“明天实际会出现的卡”

这就不是普通导出，而是调度预测。


### 难点二：要尽量和 Anki 顺序一致

这意味着不能只靠简单排序：

- 不能只是按 `due` 排
- 不能只是按卡片 ID 排
- 不能只是按牌组顺序排

必须尽量借助 Anki 自己的调度器逻辑。


### 难点三：卡片内容本身可能很复杂

很多 Anki 卡片不是纯文字，还可能包含：

- 图片
- 音频占位
- 模板 CSS
- JavaScript
- LaTeX / MathJax 公式
- 加密模板

所以导出 PDF 本质上不是“写文本到纸上”，而是“复刻一部分 Anki 前端渲染环境”。


## 8. 这份代码里最值得关注的技术点

如果以后你要继续开发或交给别人维护，最应该关注的是这几块：

### 8.1 数据库快照

关键词：

- `sqlite3.backup()`
- temporary collection

意义：
这是安全边界，决定是否会碰坏正式库。


### 8.2 调度器队列优先、SQL 兜底

关键词：

- `get_queued_cards()`
- fallback ordering

意义：
这是“顺序准确性”的核心。


### 8.3 WebView 渲染 PDF

关键词：

- `AnkiWebView`
- `stdHtml(...)`
- `printToPdf()`

意义：
这是“显示 fidelity（还原度）”的核心。


### 8.4 MathJax 与模板脚本等待

关键词：

- `MathJax.typesetPromise()`
- bridge command
- async rendering

意义：
这是“公式、加密卡、脚本卡能不能正常显示”的关键。


## 9. 未来如果继续迭代，可以优化什么？

下面这些属于“下一阶段优化”，不是当前必须项。

### 9.1 增加配置文件

让这些选项可保存：

- 每页题数
- 留白高度
- 是否显示答案附录
- 默认导出牌组


### 9.2 增加回写模式

目前纸上记录 `重来 / 困难 / 良好 / 简单` 还是手动回到 Anki 同步。
以后可以考虑做“批量回写”。

但这会显著提高风险，因为它会真正修改复习状态。


### 9.3 增加自动化测试样例

比如固定几类卡：

- 普通正反卡
- Cloze 卡
- 公式卡
- 图片卡
- 加密模板卡

每次改完插件就跑回归检查。


## 10. 一句话总结

这个插件本质上是一个：

“基于 Python + Anki API + SQLite 快照 + WebView 渲染的导出工具”

它的核心思路不是“简单打印牌组”，而是：

“先在临时数据库里模拟明天，再尽量按 Anki 实际调度顺序取卡，最后用网页方式渲染成适合打印的 PDF。”


## 11. 给非技术用户的最简短理解

如果只看一句最通俗的话：

这个插件做的事情，就是“提前把 Anki 明天会给你出的题，按大体一致的顺序排好版，打印出来做纸面练习”。
